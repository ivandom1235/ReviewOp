# Project II — Revised Novelty Plan and Implementation Blueprint

## 1. Why this revised plan exists

This document replaces the earlier novelty blueprint with a narrower and more defensible plan.

The previous version had the right direction, but it tried to claim too many things at once. This revision keeps the strongest parts and reduces the parts that are likely to be attacked in review.

The central question is now:

**How do we turn the current Project II codebase into a publishable paper centered on ambiguity-aware, evidence-grounded implicit aspect detection, without overclaiming open-world capability or overloading the paper with too many equal contributions?**

This revised plan focuses on:
- what the real paper contribution should be
- what should be primary vs secondary
- what the current code already supports
- what must change in code and evaluation
- what should be left for later versions
- a practical implementation order

---

## 2. Revised research direction

### Main paper direction

The paper should now be framed as:

> **Existing implicit ABSA systems usually force a single fixed-label prediction and under-handle ambiguity, abstention, and evidence-grounded interpretation. We introduce an ambiguity-aware, evidence-grounded benchmark and selective inference framework for implicit aspect detection, evaluated under grouped and domain-shift settings.**

This is the core claim.

### What changed from the earlier version

The earlier version tried to combine all of the following into one main novelty claim:
- ambiguity-aware
- open-world
- evidence-grounded
- selective inference
- domain shift
- ReviewOps deployment

That was too broad.

This revised version keeps:
- ambiguity-aware labeling
- evidence-grounded implicit interpretation
- selective inference with abstention
- grouped and domain-aware evaluation

It reduces or postpones:
- strong open-world claims
- graph as main novelty
- full system/deployment contribution as equal headline novelty

---

## 3. What the paper should primarily contribute

The paper should revolve around **two main contributions**, not three equal ones.

### Contribution 1 — Benchmark contribution
A new benchmark/task setting for implicit aspect detection where:
- more than one interpretation may be acceptable
- each accepted interpretation must be tied to evidence
- abstention is allowed when evidence is insufficient or ambiguity is too high
- evaluation is performed under grouped and domain-aware split settings

### Contribution 2 — Method contribution
A selective evidence-grounded inference framework that:
- scores candidate implicit aspects
- links candidates to evidence
- decides whether to answer, return multiple labels, or abstain
- is evaluated with flexible ambiguity-aware scoring rather than only exact single-label accuracy

### What is not a main contribution anymore
These should no longer be sold as equal headline novelties:
- graph normalization as core novelty
- ReviewOps dashboard as core novelty
- full open-world novel-aspect discovery as a guaranteed main contribution

These remain useful as:
- downstream application context
- system integration value
- future extension

## 6. Main research gaps this version addresses

This revised novelty plan should clearly target the following gaps.

### Gap A — Forced single-label prediction
Many implicit ABSA systems still force one answer even when the text plausibly supports more than one interpretation.

### Gap B — Weak evidence grounding
Many systems output hidden aspects without explicitly linking them to supporting evidence in the review.

### Gap C — Lack of abstention
Many systems answer even when the evidence is weak, underspecified, or genuinely ambiguous.

### Gap D — Evaluation rigidity
Many evaluations assume exact single-label correctness instead of allowing flexible scoring against multiple acceptable interpretations.

### Gap E — Weak robustness testing
Random splits are often too easy. Grouped and domain-aware splits provide a more realistic test of generalization.

These are the real backbone gaps for the revised paper.

## 8. Revised methodology

The final methodology should now have **five main stages**.

# Stage 1 — Explicit anchor extraction

Purpose:
- identify directly mentioned aspects
- extract explicit evidence spans
- provide anchor information for downstream interpretation

This is supportive infrastructure, not the main novelty.

### Input
Raw review text

### Output
A list of explicit aspect objects such as:

```json
{
  "aspect": "battery life",
  "sentiment": "negative",
  "evidence_span": [13, 38],
  "evidence_text": "dies before evening",
  "source": "explicit"
}
```

---

# Stage 2 — Implicit candidate scoring

Purpose:
Generate and score plausible hidden aspect candidates for the review.

This stage may use:
- retrieval from similar implicit examples
- prototype-based scoring
- encoder-based ranking
- weak supervision
- explicit↔implicit paraphrase alignment

Important framing:
This is **candidate scoring**, not yet forced final prediction.

### Input
Review text and optional domain metadata

### Output
A candidate list such as:

```json
[
  {"candidate_aspect": "battery life", "score": 0.74},
  {"candidate_aspect": "power efficiency", "score": 0.58},
  {"candidate_aspect": "charging frequency", "score": 0.51}
]
```

---

# Stage 3 — Evidence linking

Purpose:
Every accepted implicit interpretation must be linked to explicit support in the text.

The system should locate:
- supporting sentence
- supporting span when possible
- support type
- evidence confidence

### Input
Review text + candidate aspects

### Output
```json
{
  "candidate_aspect": "battery life",
  "evidence_text": "I had to recharge it twice before dinner",
  "evidence_span": [0, 41],
  "support_type": "symptom_based",
  "evidence_confidence": 0.82
}
```

### Important definition
For this paper, **evidence-grounded** means:
- every accepted implicit prediction must include a supporting sentence or span,
- the evidence must be explicitly attached to the interpretation,
- unsupported predictions must be rejected or abstained.

---

# Stage 4 — Selective decision layer

Purpose:
The model should not always guess.

The final decision should be one of:
- return one label
- return more than one acceptable label
- abstain

### Input
Candidate scores + evidence confidence + ambiguity signal

### Output
```json
{
  "decision": "abstain",
  "reason": "insufficient_evidence",
  "confidence": 0.41,
  "ambiguity_score": 0.67
}
```

or

```json
{
  "decision": "multi_label",
  "labels": ["battery life", "power efficiency"],
  "confidence": 0.76,
  "ambiguity_score": 0.71
}
```

### Important note
This selective layer is a key method contribution.
It should be presented as a practical decision framework, not as an overly theoretical novelty claim.

---

# Stage 5 — Benchmark-aware evaluation and downstream export

Purpose:
Evaluate predictions under flexible scoring and prepare accepted outputs for downstream ReviewOps use.

This stage includes:
- flexible correctness against multiple acceptable labels
- abstention-aware evaluation
- grouped and domain-aware split evaluation
- export of accepted predictions for analytics/backend use

Graph and dashboard integration may consume accepted outputs later, but they should not define the main scientific novelty.

---

## 10. Revised formulas

The formulas should stay simple and implementable.

# 10.1 Prototype scoring

If a prototype head is used, class prototype for label \(c\):

\[
p_c = \frac{1}{|S_c|} \sum_{x_i \in S_c} f_\theta(x_i)
\]

Query score using cosine similarity:

\[
s(q, c) = \cos(f_\theta(q), p_c)
\]

Probability over labels:

\[
P(c \mid q) = \frac{\exp(s(q,c)/\tau)}{\sum_{c'} \exp(s(q,c')/\tau)}
\]

---

# 10.2 Ambiguity score

A practical ambiguity score can be:

\[
A(q) = 1 - (p_1 - p_2)
\]

where:
- \(p_1\) is the top probability
- \(p_2\) is the second-highest probability

If the top two probabilities are close, ambiguity is high.

Later versions may enrich this with annotator disagreement or conformal set size.

---

# 10.3 Selective decision rule

Let:
- \(p_{max}\) = maximum label probability
- \(e\) = evidence quality score
- \(a\) = ambiguity score

A simple practical confidence score can be:

\[
C(q) = \alpha p_{max} + \beta e - \gamma a
\]

Decision rule:
- if \(C(q) < T_{abstain}\), abstain
- else if two or more labels are within margin \(m\), return multi-label
- else return top label

### Important revision
This paper should not overemphasize the formula itself as the novelty.
It is a practical selective decision layer.

---

# 10.4 Flexible benchmark matching

Standard exact match is too strict.

For prediction set \(\hat{Y}\) and gold interpretation set \(Y^*\):

\[
\text{FlexCorrect}(\hat{y}) = 1 \; \text{if} \; \hat{y} \in Y^*
\]

For multiple predictions, use set overlap such as Jaccard:

\[
J(\hat{Y}, Y^*) = \frac{|\hat{Y} \cap Y^*|}{|\hat{Y} \cup Y^*|}
\]

---

# 10.5 Calibration

Expected Calibration Error:

\[
ECE = \sum_{b=1}^{B} \frac{|B_b|}{n} |acc(B_b) - conf(B_b)|
\]

This remains useful because selective inference depends on confidence quality.

# 11.2 reviewlevel_adapter — required changes

### Required fixes
- preserve evidence spans when available
- stop falling back to full review text unless explicitly marked as fallback
- preserve parent identity more strictly
- preserve whether the row came from ambiguous or single-gold annotation

This is important because evidence-grounded experiments become weak if the adapter collapses evidence to the whole review.

---

### A. Reposition ProtoNet
ProtoNet should remain:
- a baseline,

### B. Add candidate-scoring behavior
The implicit model should expose candidate scores rather than only final joint-label predictions.

### C. Add selective decision logic
Support:
- answer
- multi-label answer
- abstain

### D. Preserve true evidence usage
Inference should consume true evidence spans or evidence sentences whenever possible.

### E. Add benchmark-aware evaluation support
Evaluation should support:
- any acceptable gold interpretation
- multi-label overlap
- abstention correctness
- ambiguity-sliced performance

### F. Keep joint aspect+sentiment as baseline only
Joint label prediction can remain as one baseline setting, but it should not be the main task definition.

## 15. Risks to avoid

### Risk 1 — Claiming too many novelties at once
Do not headline ambiguity, open-world, graph, LLM, ReviewOps, and domain shift as equal novelties.

### Risk 2 — Using “open-world” too early
Unless novel-aspect handling is strongly implemented and evaluated, avoid presenting the first paper as full open-world implicit discovery.

### Risk 3 — Letting ProtoNet define the paper
ProtoNet is a useful baseline/component, but it should not define the novelty claim.

### Risk 4 — Weak evidence preservation
If evidence collapses to full review text in the adapter, the evidence-grounded claim becomes weak.

### Risk 5 — Random split only
Grouped and domain-aware evaluation are necessary for a stronger paper.

### Risk 6 — Mixing system contribution with main research contribution
ReviewOps integration is useful, but only treat it as main novelty if it is experimentally validated.
