# Project II — Novelty Blueprint and Implementation Document

## 1. Why this document exists

This document converts the current Project II codebase into a clear research and implementation plan.

It is written to answer one main question:

**How do we move from the current ProtoNet + dataset builder starting point to a truly novel paper built around ambiguity-aware, open-world, evidence-grounded implicit aspect detection under domain shift?**

This document covers:
- what currently exists
- what the current code is actually doing
- what the real research goal should be
- what gaps remain
- what needs to change in code
- the methodology to implement
- formulas and expected inputs/outputs
- a practical coding roadmap
- simple questions that still need answers

---

## 2. Current research direction

### Current target novelty

The research direction we are moving toward is:

> **Current ABSA benchmarks and systems still under-handle ambiguous implicit aspect interpretation. We introduce an ambiguity-aware, open-world, evidence-grounded benchmark and selective inference framework for implicit aspect detection under domain shift.**

This means the paper should no longer be framed as only:
- explicit extraction
- implicit ProtoNet classification
- graph building
- dashboard analytics

Instead, the paper should be framed as:

1. a **new benchmark/task setting**
2. a **selective evidence-grounded inference framework**
3. a **cross-domain ReviewOps deployment story**

# 3.3 What the current ProtoNet code does

The current ProtoNet codebase gives a clean baseline training shell.

It supports:
- `train`
- `eval`
- `export`
- `episodic` input mode
- `reviewlevel` input mode
- transformer or BOW encoder backends
- checkpointing
- prediction export
- report generation
- global prototype bank export

### What it is really classifying right now

The current ProtoNet task is effectively:

**joint label classification = aspect + sentiment**

That means labels currently look like:
- `battery life__negative`
- `service quality__positive`
- etc.

### What the current adaptation path does

The review-level adapter takes rows with labels and expands them into examples with fields such as:
- `example_id`
- `parent_review_id`
- `review_text`
- `evidence_sentence`
- `aspect`
- `implicit_aspect`
- `sentiment`
- `label_type`
- `confidence`

### Important strength already present

The episode builder already attempts to avoid support/query leakage **inside each episode** by:
- deduplicating by `parent_review_id`
- rejecting overlap between support and query IDs

This is good.

### Important current limitation

The current ProtoNet is still being used as a **closed-set classifier shell**.

For the target novelty, that is not enough.

It needs to move toward:
- candidate scoring
- selective acceptance
- abstention
- known-vs-novel routing
- evidence-aware decision making

ProtoNet should become:
- a baseline
- or a component of the implicit branch

It should **not** remain the whole research story.

## 5. Core research claim and contribution structure

The final paper should revolve around 3 contributions.

### Contribution 1 — Benchmark / task contribution
A new benchmark/task for implicit aspect detection where:
- more than one hidden aspect can be valid
- the model may abstain
- the model may output a novel aspect candidate
- each accepted label must be tied to evidence
- evaluation is done under random, source-holdout, and domain-holdout settings

### Contribution 2 — Method contribution
A selective inference framework that:
- generates implicit candidates
- links them to evidence
- decides known vs novel
- decides answer vs abstain
- optionally returns multiple valid labels
- passes accepted outputs to graph normalization and ReviewOps analytics

### Contribution 3 — System contribution
An operational ReviewOps pipeline where:
- explicit + implicit outputs coexist
- accepted predictions are evidence-backed
- abstained cases are visible
- novel issues can be surfaced
- graph/grouping helps analytics and alerts

## 6. Main gaps we are trying to resolve

The current novelty target mainly addresses these unresolved gaps:

### Gap A — Implicit aspect ambiguity is under-modeled
Current systems still usually force one answer.
We need support for multiple valid hidden interpretations.

### Gap B — Open-world implicit detection is underdeveloped
Current pipelines usually assume a fixed inventory.
We need known-vs-novel behavior.

### Gap C — Evidence-grounded abstention is missing
Current systems usually always answer.
We need answer / multi-answer / abstain behavior.

### Gap D — Cross-domain robustness is not solved cleanly
We need random, source-holdout, and domain-holdout evaluation.

### Gap E — Annotation and evaluation are too rigid
We need multiple gold interpretations and flexible scoring.

These gaps should become the real backbone of the paper.

## 7. Proposed final methodology

The final methodology should have 6 stages.

# Stage 1 — Explicit anchor extraction

Purpose:
- find directly mentioned aspects
- find explicit evidence
- help graph construction
- support downstream grounding

This is not the headline novelty, but it is an important anchor layer.

### Input
Raw review text

### Output
A list of explicit aspect objects such as:
```json
{
  "aspect": "battery life",
  "sentiment": "negative",
  "evidence_span": [13, 38],
  "evidence_text": "dies before evening",
  "source": "explicit"
}
```

---

# Stage 2 — Implicit candidate generation

Purpose:
Generate possible hidden aspects from the review text.

This can use:
- latent-aspect rules
- retrieval from similar examples
- encoder-based ranking
- weak supervision
- explicit↔implicit paraphrase knowledge
- LLM proposal generation only where needed

### Input
Review text and optional domain metadata

### Output
A candidate list such as:
```json
[
  {"candidate_aspect": "battery life", "score": 0.74},
  {"candidate_aspect": "power efficiency", "score": 0.58},
  {"candidate_aspect": "charging frequency", "score": 0.51}
]
```

---

# Stage 3 — Evidence linking

Purpose:
Every accepted hidden aspect must be supported by text.

The system should locate:
- supporting sentence
- supporting span
- support type
- confidence

### Input
Review text + candidate aspects

### Output
```json
{
  "candidate_aspect": "battery life",
  "evidence_text": "I had to recharge it twice before dinner",
  "evidence_span": [0, 41],
  "support_type": "near_exact",
  "evidence_confidence": 0.82
}
```

---

# Stage 4 — Known-vs-novel routing

Purpose:
Decide whether the candidate belongs to:
- a known canonical implicit aspect
- a novel aspect cluster
- or should be rejected

### Input
Candidate + evidence + canonical latent inventory

### Output
```json
{
  "routing": "known",
  "aspect_label": "battery life",
  "novel_score": 0.11
}
```

or

```json
{
  "routing": "novel",
  "aspect_label": "charging inconvenience",
  "novel_score": 0.81
}
```

---

# Stage 5 — Selective prediction / abstention

Purpose:
The model should not always guess.

The final decision should be one of:
- answer with one label
- answer with multiple labels
- abstain

### Input
Candidate scores + evidence confidence + novelty score + ambiguity signal

### Output
```json
{
  "decision": "abstain",
  "reason": "insufficient_evidence",
  "confidence": 0.41,
  "ambiguity_score": 0.67
}
```

or

```json
{
  "decision": "multi_label",
  "labels": ["battery life", "power efficiency"],
  "confidence": 0.76,
  "ambiguity_score": 0.71
}
```

---

# Stage 6 — Graph normalization and ReviewOps integration

Purpose:
Use accepted outputs for:
- aspect normalization
- grouping and clustering
- trend analysis
- alerting
- dashboard summaries
- evidence drill-down

The graph is a systems and analytics strength, but should not be sold as the main novelty.

## 9. Recommended formulas

These formulas do not need to be mathematically exotic. They need to be correct, explainable, and implementable.

# 9.1 Prototype classification

If ProtoNet or a prototype head is used, class prototype for label \(c\):

\[
p_c = \frac{1}{|S_c|} \sum_{x_i \in S_c} f_\theta(x_i)
\]

where:
- \(S_c\) = support examples of class \(c\)
- \(f_\theta(x_i)\) = encoded embedding

Query classification score using cosine similarity:

\[
s(q, c) = \cos(f_\theta(q), p_c)
\]

Probability over labels:

\[
P(c \mid q) = \frac{\exp(s(q,c)/\tau)}{\sum_{c'} \exp(s(q,c')/\tau)}
\]

where \(\tau\) is temperature.

---

# 9.2 Selective prediction / abstention

Let:
- \(p_{max}\) = maximum label probability
- \(e\) = evidence quality score
- \(n\) = novelty score
- \(a\) = ambiguity score

A simple selective confidence score can be:

\[
C(q) = \alpha p_{max} + \beta e - \gamma a - \delta n
\]

Then:
- if \(C(q) < T_{abstain}\), abstain
- else if two or more labels are within margin \(m\), return multi-label
- else return top known label

This is simple and implementable.

---

# 9.3 Ambiguity score

A practical ambiguity score can be based on closeness of top-k probabilities:

\[
A(q) = 1 - (p_1 - p_2)
\]

where:
- \(p_1\) = top probability
- \(p_2\) = second-highest probability

If the top two probabilities are close, ambiguity is high.

You can later enrich this using annotator disagreement or conformal set size.

---

# 9.4 Novelty score

If using prototype distance:

\[
N(q) = 1 - \max_c \cos(f_\theta(q), p_c)
\]

If novelty score is above threshold, route to novel-aspect candidate handling.

---

# 9.5 Flexible benchmark matching

Standard exact match is too strict.
For a prediction set \(\hat{Y}\) and gold interpretation set \(Y^*\), flexible correctness can be:

\[
\text{FlexCorrect}(\hat{y}) = 1 \; \text{if} \; \hat{y} \in Y^*
\]

For multiple predictions, use Jaccard or set overlap:

\[
J(\hat{Y}, Y^*) = \frac{|\hat{Y} \cap Y^*|}{|\hat{Y} \cup Y^*|}
\]

---

# 9.6 Calibration / confidence quality

Expected Calibration Error:

\[
ECE = \sum_{b=1}^{B} \frac{|B_b|}{n} |acc(B_b) - conf(B_b)|
\]

This is already aligned with your current ProtoNet evaluator style.

This is the most important section for actual implementation.

# 11.1 dataset_builder — what must change

### A. Add a benchmark export layer
Create a new export path:

### B. Add benchmark schema classes
Add dataclasses or typed schemas for:
- `GoldInterpretation`
- `BenchmarkInstance`
- `SplitProtocolAssignment`

### C. Separate training export from benchmark export
Do not reuse the same exact row objects for both.
Training rows can remain model-friendly.
Benchmark rows must remain paper-friendly.

### D. Add grouped split support
Current splitting looks row-level/random.
Need grouped split options by something like:
- parent review family
- product/business ID
- synthetic parent ID
- source cluster

### E. Add ambiguity-aware annotation merge logic
Extend `gold_annotations` support so it can merge:
- multiple valid labels
- annotator support counts
- abstain flags
- novel flags

### F. Add benchmark diagnostics
Need new report sections for:
- ambiguous instance rate
- average gold interpretations per instance
- abstain-acceptable rate
- novel-acceptable rate
- grouped split leakage diagnostics

---

# 11.2 reviewlevel_adapter — what must change

The current adapter converts labels into single examples.
That is fine for training, but not for the new benchmark.

### Required changes
- preserve full evidence span when available
- preserve whether the label came from ambiguous gold or single gold
- optionally carry multiple candidate labels for benchmark-aware evaluation
- preserve parent identity and source family more strictly

### Important note
The current path often falls back to full review text as `evidence_sentence`.
This must be fixed for serious implicit evidence-grounded experiments.

---

# 11.3 ProtoNet / implicit model — what must change

### A. Do not keep only joint-label classification as the main task
Joint label prediction can remain as:
- a baseline
- a secondary metric

But the main task should become:
- implicit aspect selective inference
- sentiment either secondary or post-classification

### B. Add answer / abstain / multi-label decision logic
This can be implemented after raw class scoring.

### C. Add known-vs-novel routing
Use distance-to-prototype or other novelty criteria.

### D. Preserve true evidence
The model should consume actual evidence spans or evidence sentences, not always full review text.

### E. Support benchmark-aware evaluation
Evaluation should check against:
- any acceptable gold label
- multi-label correctness
- abstention correctness
- novelty correctness

### F. Composite model selection
Checkpoint selection should not depend on only validation accuracy.
Use a weighted objective over:
- flexible macro-F1
- selective accuracy
- domain-holdout performance
- calibration
- known-vs-novel routing quality

---

# 11.4 evaluator — what must change

Current evaluator already computes:
- accuracy
- aspect-only accuracy
- macro-F1
- ECE
- low-confidence rate

Needed additions:
- flexible match score
- multi-label overlap score
- abstention precision/recall
- coverage-risk metrics
- known-vs-novel AUROC or F1
- ambiguity-sliced evaluation
- domain/source holdout breakdowns

---

# 11.5 ReviewOps backend / graph — what must change

The backend/UI side should support new inference states:
- accepted prediction
- abstained prediction
- novel aspect candidate
- ambiguous multi-label prediction

Graph layer should only consume **accepted** predictions by default.
Optional separate graph can track novel candidates.

## 18. Final summary

### What we currently have
- a good dataset builder starting point
- a working ProtoNet training shell
- strict filtering and reporting logic
- explicit and implicit system pieces already connected
- a ReviewOps architecture that can use the outputs

### What we currently do not have
- a real ambiguity-aware benchmark artifact
- abstention-aware evaluation
- open-world novel-aspect handling
- grouped split rigor
- fully preserved evidence into ProtoNet
- a single paper claim that unifies all modules

### What we need to do
- turn dataset_builder into a benchmark constructor
- turn implicit inference into a selective answer/abstain framework
- make ProtoNet a component/baseline, not the whole story
- preserve evidence end to end
- evaluate under real domain shift
- let ReviewOps consume accepted, abstained, and novel outputs separately

That is how the current codebase can be transformed into the intended novel paper.
